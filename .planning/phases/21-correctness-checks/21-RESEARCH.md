# Phase 21: Correctness Checks and Challenge Definitions - Research

**Researched:** 2026-03-11
**Domain:** Behavioral evaluation scripts, declarative challenge config, reference solutions, git tagging
**Confidence:** HIGH

## Summary

Phase 21 bridges the challenge project (Phase 20) to the benchmark harness (Phase 22) by building the evaluation infrastructure: per-challenge correctness check scripts, declarative JSON challenge definitions, reference solutions for validation, and the `bench/after-delete` git tag. This is still a zero-API-token phase -- everything is hand-authored Bash and JSON, not generated by Claude.

The critical insight is that correctness checks must be **behavioral, not structural**. They must test observable outcomes (running `taskctl done 3` and verifying task 3 is marked done) rather than inspecting code patterns (grepping for `select(.id ==`). This is explicitly required by HARN-03 and essential for fair benchmarking -- different AI modes may produce valid solutions with different implementation approaches.

Each correctness check script must pass two controls: it MUST fail against `bench/baseline` (negative control, proving the check detects real deficiencies) and MUST pass against a hand-written reference solution (positive control, proving the check accepts valid fixes). These dual controls are the core success criteria for this phase.

**Primary recommendation:** Build 5 check scripts under `benchmarks/challenges/checks/`, 5 JSON definition files under `benchmarks/challenges/`, and 5 reference solution branches/directories. Validate each check against both controls before committing. The `bench/after-delete` tag is created by applying the Challenge 2 reference solution to the baseline.

### Pattern 1: Behavioral Check Script

**What:** Each check script receives the taskctl project directory as an argument (or uses the current working directory), runs commands/tests, and outputs a structured pass/fail per check. Returns 0 if all pass, 1 if any fail.
**When to use:** Called by `bench-eval.sh` after a benchmark run completes.
**Example:**
```bash
#!/bin/bash
# checks/check-fix-bug.sh -- Challenge 1 correctness check
# Behavioral checks ONLY -- tests outcomes, not code patterns
set -euo pipefail

TASKCTL_DIR="${1:-.}"
BATS_BIN="${2:-../../tests/bats/bin/bats}"
SCRIPT_DIR="$TASKCTL_DIR/src"
passed=0
failed=0
total=0

check() {
    local name="$1"
    total=$((total + 1))
    if eval "$2"; then
        echo "PASS: $name"
        passed=$((passed + 1))
    else
        echo "FAIL: $name"
        failed=$((failed + 1))
    fi
}

# Check 1: taskctl done 3 marks task 3 (not task 4)
check "done 3 marks task with id=3" '
    local tmpdata
    tmpdata=$(mktemp /tmp/taskctl_check_XXXX.json)
    cp "$TASKCTL_DIR/.taskctl.json" "$tmpdata"
    TASKCTL_DATA="$tmpdata" bash -c "
        source \"$SCRIPT_DIR/storage.sh\"
        source \"$SCRIPT_DIR/format.sh\"
        source \"$SCRIPT_DIR/commands/done.sh\"
        cmd_done 3
    " >/dev/null 2>&1
    local task3_done
    task3_done=$(jq -r ".[] | select(.id == 3) | .done" "$tmpdata")
    rm -f "$tmpdata"
    [[ "$task3_done" = "true" ]]
'

# Check 2: Existing tests pass
check "existing tests pass" '
    (cd "$TASKCTL_DIR" && "$BATS_BIN" tests/ >/dev/null 2>&1)
'

# Check 3: New/modified test covers the fix
check "test coverage for done command exists" '
    local test_files
    test_files=$(find "$TASKCTL_DIR/tests" -name "*.bats" -newer "$TASKCTL_DIR/src/commands/done.sh" 2>/dev/null || true)
    # OR: test_done.bats exists, or existing test files were modified
    [[ -f "$TASKCTL_DIR/tests/test_done.bats" ]] || [[ -n "$test_files" ]]
'

echo ""
echo "Score: $passed/$total checks passed"
exit $((failed > 0 ? 1 : 0))
```

### Pattern 2: Declarative Challenge JSON

**What:** Each challenge has a JSON definition file with all metadata the harness needs.
**When to use:** `bench-eval.sh` and `bench-run.sh` read these to configure execution.
**Schema:**
```json
{
  "id": "fix-bug",
  "name": "Fix the Bug",
  "number": 1,
  "starting_tag": "bench/baseline",
  "prompt": "The `taskctl done 3` command marks the wrong task as done. Find the bug and fix it.",
  "time_cap_minutes": 10,
  "check_script": "checks/check-fix-bug.sh",
  "checks": [
    "done 3 marks task with id=3",
    "existing tests pass",
    "test coverage for done command exists"
  ],
  "measures": "Diagnostic reasoning, targeted fix quality, regression avoidance"
}
```

### Pattern 3: bench-eval.sh Driver

**What:** A wrapper script that loads the challenge JSON, resolves the check script path, runs it, and outputs structured results.
**When to use:** Called by `bench-run.sh` (Phase 22) and by developers validating checks.
**Example:**
```bash
#!/bin/bash
# harness/bench-eval.sh -- Run correctness checks for a challenge
set -euo pipefail

CHALLENGE="${1:?Usage: bench-eval.sh <challenge-name> [taskctl-dir]}"
TASKCTL_DIR="${2:-benchmarks/taskctl}"
HARNESS_DIR="$(cd "$(dirname "${BASH_SOURCE[0]}")" && pwd)"
CHALLENGES_DIR="$(cd "$HARNESS_DIR/../challenges" && pwd)"

# Load challenge definition
CHALLENGE_FILE="$CHALLENGES_DIR/$CHALLENGE.json"
if [[ ! -f "$CHALLENGE_FILE" ]]; then
    echo "Error: Unknown challenge '$CHALLENGE'" >&2
    echo "Available: $(ls "$CHALLENGES_DIR"/*.json 2>/dev/null | xargs -I{} basename {} .json | tr '\n' ' ')" >&2
    exit 1
fi

CHECK_SCRIPT=$(jq -r '.check_script' "$CHALLENGE_FILE")
CHALLENGE_NAME=$(jq -r '.name' "$CHALLENGE_FILE")

echo "=== Evaluating: $CHALLENGE_NAME ==="
echo ""

# Run the check script
bash "$CHALLENGES_DIR/$CHECK_SCRIPT" "$TASKCTL_DIR"
exit_code=$?

echo ""
if [[ $exit_code -eq 0 ]]; then
    echo "RESULT: PASS"
else
    echo "RESULT: FAIL"
fi
exit $exit_code
```

### Pattern 4: Reference Solution as Overlay

**What:** Each reference solution directory contains only the files that differ from the baseline. During validation, these files are copied over the baseline to create the "correct" state.
**When to use:** Positive control validation of check scripts.
**Example:**
```
reference-solutions/fix-bug/
  src/commands/done.sh          # Fixed done.sh (looks up by .id field)
  tests/test_done.bats          # New test covering the fix
```

### Anti-Patterns to Avoid

- **Structural checks:** Never `grep 'select(.id' src/commands/done.sh`. Different solutions may use `map(if .id ==` or `(.[] | select(.id ==` or a completely different approach. Test the behavior: run the command and check the output/data.
- **Checking file timestamps:** Unreliable in git worktrees. Use file existence and content checks instead.
- **Hardcoded paths in check scripts:** Always accept the taskctl directory as an argument. The harness runs checks in isolated worktrees.
- **Forgetting to test negative control:** A check that always passes is worthless. Every check MUST fail against `bench/baseline`.
- **Overly strict checks:** Don't check exact output format. Check that the right data is in the JSON, not the exact printf string.
- **Running checks in the real repo:** Always use temp copies or worktree isolation to avoid corrupting the baseline.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Running Bats tests | Custom test runner | `bats tests/*.bats` | Bats already handles test discovery, temp dirs, assertions |
| JSON assertion | String matching on JSON output | `jq` queries with specific field checks | Resilient to formatting changes, whitespace |
| ShellCheck analysis | Custom code quality metrics | `shellcheck --format=json` or `shellcheck --format=gcc` + grep -c | Deterministic, well-defined warning categories |
| Line count/diff measurement | Custom diff parser | `diff --stat` or `wc -l` | Standard, reliable |
| Temp file management | Manual mkdir/cleanup | `mktemp -d` + trap cleanup | Handles interrupts, avoids leaks |

**Key insight:** Check scripts should be thin wrappers around existing tools (Bats, jq, shellcheck, diff). The evaluation logic is in the orchestration, not in reimplementing what these tools already do.

## Common Pitfalls

### Pitfall 1: Check That Passes on Baseline (False Positive)

**What goes wrong:** A check script returns PASS against `bench/baseline`, meaning it cannot distinguish fixed from unfixed code.
**Why it happens:** The check tests something that is already true at baseline (e.g., "storage.sh exists" instead of "test_storage.bats exists and passes").
**How to avoid:** For every check script, run it against `bench/baseline` FIRST and verify it returns FAIL. This is the negative control requirement.
**Warning signs:** Check passes without applying any solution.

### Pitfall 2: Check That Fails on Valid Solution (False Negative)

**What goes wrong:** A check script returns FAIL even when the reference solution is applied, because the check is too strict or structurally coupled.
**Why it happens:** Checking for exact file names (e.g., requiring `test_done.bats` when the solver might add tests to an existing file), checking for exact output strings, or checking code structure.
**How to avoid:** Test outcomes, not implementations. For "new tests exist," check that there are more test files or more test functions than at baseline, not that a specific filename exists.
**Warning signs:** Reference solution fails the check.

### Pitfall 3: bench/after-delete Tag Out of Sync

**What goes wrong:** The `bench/after-delete` tag doesn't include a properly working delete command, or it includes other changes that contaminate Challenge 5's starting state.
**Why it happens:** Applying the Challenge 2 reference solution carelessly, or tagging before verifying the delete command works.
**How to avoid:** Create `bench/after-delete` by: (1) checkout `bench/baseline`, (2) apply ONLY the Challenge 2 reference files (delete.sh + test_delete.bats + taskctl.sh dispatch), (3) verify delete works and all existing tests still pass, (4) tag.
**Warning signs:** Challenge 5 checks fail because the starting state is wrong.

### Pitfall 4: Check Scripts Not Portable to Worktrees

**What goes wrong:** Check scripts work when run from the repo root but fail in isolated worktrees because they reference paths relative to the git root.
**Why it happens:** Phase 22 runs each benchmark in a fresh git worktree. Paths like `../../tests/bats/bin/bats` may not resolve.
**How to avoid:** Check scripts should accept the Bats binary path as a parameter or discover it dynamically. The harness should pass resolved paths.
**Warning signs:** Checks pass locally but fail in `bench-run.sh`.

### Pitfall 5: Race Conditions in Check Scripts

**What goes wrong:** Multiple check assertions modify the same data file, causing later assertions to see corrupted state.
**Why it happens:** All checks use the same `.taskctl.json` without isolation.
**How to avoid:** Each behavioral check that modifies data should copy `.taskctl.json` to a temp file, operate on the copy, then clean up. Never modify the working copy.
**Warning signs:** Checks pass individually but fail when run together.

### Pitfall 6: Challenge 4 Complexity Measurement Fragility

**What goes wrong:** The "cyclomatic complexity reduced" check is too coarse or too strict, passing trivially or failing valid refactors.
**Why it happens:** Bash does not have a standard cyclomatic complexity tool. Proxies (ShellCheck warnings, line count) are imperfect.
**How to avoid:** Use multiple signals: (1) `diff` shows > 10 lines changed (mechanical change happened), (2) ShellCheck warning count decreased (quality improved), (3) line count same or reduced (no bloat). Require 2 of 3, not all 3.
**Warning signs:** Valid refactoring that consolidates duplicated code fails because line count went up slightly due to added comments.

**Negative control (baseline):** Bug exists. `done 3` marks array[3] (task id=4), not the task with id=3. With seed data: task 3 remains `done: false`, task 4 becomes `done: true`.

**Behavioral checks:**
```bash

cp "$TASKCTL_DIR/.taskctl.json" "$tmpdata"

# Check 2: Existing tests still pass
(cd "$TASKCTL_DIR" && "$BATS_BIN" tests/test_add.bats tests/test_list.bats)

# Check 3: Test covers the fix (new test file or modified test)

[[ -f "$TASKCTL_DIR/tests/test_done.bats" ]] || \
```

### Challenge 4: Refactor -- Check Design

**Negative control (baseline):** format.sh has 22 ShellCheck warnings and duplicated code. Diff is 0 lines (unchanged).

**Behavioral checks:**
```bash
# Check 1: All existing tests pass
(cd "$TASKCTL_DIR" && "$BATS_BIN" tests/)

[[ "$diff_lines" -gt 10 ]]

[[ "$current_warnings" -lt "$baseline_warnings" ]]

# Check 4: No new functionality (no new functions or commands)

current_functions=$(grep -c '^\w\+()' "$TASKCTL_DIR/src/format.sh" || echo 0)
# Allow same or fewer functions (consolidation is good)
[[ "$current_functions" -le "$((baseline_functions + 1))" ]]
```

### Reference Solution: Challenge 1 (Fix Bug)

**Fixed done.sh:**
```bash
# commands/done.sh -- Mark a task as done
cmd_done() {
    local task_id="${1:-}"
    if [[ -z "$task_id" ]]; then
        echo "Error: task ID required" >&2
        exit 1
    fi
    local tasks
    tasks=$(storage_read_all)
    # Look up by .id field, not array index
    local found
    found=$(echo "$tasks" | jq --argjson id "$task_id" '[.[] | select(.id == $id)] | length')
    if [[ "$found" -eq 0 ]]; then
        echo "Error: task $task_id not found" >&2
        exit 1
    fi
    local updated
    updated=$(echo "$tasks" | jq --argjson id "$task_id" \
        'map(if .id == $id then .done = true else . end)')
    echo "$updated" > "$STORAGE_FILE"
    echo "Task $task_id marked as done"
}
```

### Reference Solution: Challenge 2 (Add Delete Command)

**New delete.sh:**
```bash
# commands/delete.sh -- Delete a task by ID
cmd_delete() {
    local task_id="${1:-}"
    if [[ -z "$task_id" ]]; then
        echo "Error: task ID required" >&2
        exit 1
    fi
    local tasks
    tasks=$(storage_read_all)
    local found
    found=$(echo "$tasks" | jq --argjson id "$task_id" '[.[] | select(.id == $id)] | length')
    if [[ "$found" -eq 0 ]]; then
        echo "Error: task $task_id not found" >&2
        exit 1
    fi
    local updated
    updated=$(echo "$tasks" | jq --argjson id "$task_id" \
        '[.[] | select(.id != $id)]')
    echo "$updated" > "$STORAGE_FILE"
    echo "Task $task_id deleted"
}
```

**Updated taskctl.sh dispatch (for bench/after-delete):**
```bash
case "${1:-}" in
    add)    shift; source "$SCRIPT_DIR/commands/add.sh"; cmd_add "$@" ;;
    list)   shift; source "$SCRIPT_DIR/commands/list.sh"; cmd_list "$@" ;;
    done)   shift; source "$SCRIPT_DIR/commands/done.sh"; cmd_done "$@" ;;
    delete) shift; source "$SCRIPT_DIR/commands/delete.sh"; cmd_delete "$@" ;;
    *)      echo "Usage: taskctl {add|list|done|delete} [args]"; exit 1 ;;
esac
```

## bench/after-delete Tag Creation Strategy

The `bench/after-delete` tag represents the state where Challenge 2 (add delete command) has been completed. This serves as the starting state for Challenge 5 (priority feature).

**Creation steps:**
1. Create a temporary worktree from `bench/baseline`
2. Apply the Challenge 2 reference solution files: `delete.sh`, `test_delete.bats`, updated `taskctl.sh`
3. Verify: `taskctl delete 1` works, `taskctl delete 999` errors, existing tests pass, `test_delete.bats` passes
4. Commit the changes
5. Tag as `bench/after-delete` (annotated)
6. Clean up worktree

**Important:** Do NOT apply Challenge 1 (bug fix) to the after-delete state. The bug in `done.sh` should still be present at `bench/after-delete`. Challenge 5 starts from "delete exists but bug unfixed." This matches the BENCHMARK-MILESTONE.md spec: challenges are independent (except 5 depends on 2's feature).

## Scoring Architecture

Each check script should output a consistent format for `bench-eval.sh` to parse:

```
PASS: check name
FAIL: check name
...
Score: N/M checks passed
```

The eval driver computes correctness as a percentage: `(passed / total) * 100`. This maps directly to the `correctness_score` field in the result JSON schema (Phase 22).

**Scoring rules:**
- Each check has equal weight (no partial credit within a check)
- All checks must be independent (order should not matter)
- A check either passes (1) or fails (0)
- Overall score = `passed / total * 100`

## Open Questions

1. **Reference solutions: directory overlays vs. git branches?**
   - What we know: We need reference solutions for positive control. We also need `bench/after-delete` as a git tag.
   - What's unclear: Whether to store reference solutions as files in `reference-solutions/` directories or as separate git branches.
   - Recommendation: Use directory overlays for validation convenience (copy files over baseline, run checks). The `bench/after-delete` tag is created separately by applying the Challenge 2 overlay and committing. This avoids branch management complexity.

2. **Challenge 3 check: how to verify "storage.sh not modified"?**
   - What we know: The check needs to confirm storage.sh is unchanged from baseline.
   - What's unclear: In a worktree created from bench/baseline, we can use `git diff` or compare against the known baseline content.
   - Recommendation: Use `git show bench/baseline:benchmarks/taskctl/src/storage.sh` as the reference and `diff` against the current file. This works in worktrees because the tag is accessible from any worktree.

3. **Challenge 4: what counts as "no new functionality"?**
   - What we know: The check must verify refactoring, not feature addition.
