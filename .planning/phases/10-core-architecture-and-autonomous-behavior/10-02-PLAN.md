    1. **Header** -- "gsd-ralph v2.0 Architecture" with one-sentence summary: thin integration layer bridging GSD planning with Ralph autonomous execution
    2. **Core Principle** -- "Coupling to GSD tools is fine -- duplicating GSD logic is not" (exact user quote from CONTEXT.md)
    3. **What gsd-ralph Does** (CAN list):
       - Read STATE.md to determine current execution position
       - Read PLAN.md files to provide context to Claude Code
       - Read config.json for Ralph-specific settings
       - Call gsd-tools.cjs commands (coupled, not duplicated)
       - Assemble GSD context for --append-system-prompt-file injection
       - Launch headless Claude Code instances with appropriate permissions
       - Detect iteration completion/failure from exit codes
       - Loop fresh instances for incomplete work
    4. **What gsd-ralph NEVER Does** (NEVER list):
       - NEVER parse ROADMAP.md to determine phase ordering (GSD does this)
       - NEVER update STATE.md directly (GSD handles this)
       - NEVER generate plans, research, or summaries (GSD does this)
       - NEVER manage worktrees (Claude Code handles this via --worktree)
       - NEVER replicate permission logic (Claude Code's --permission-mode and --allowedTools handle this)
       - NEVER invoke GSD slash commands in headless mode (they are unavailable; use natural language prompts + system prompt injection)
    5. **Component Boundaries** -- table showing each component (SKILL.md, config schema, context assembly, launcher) with what it owns and what it delegates
    6. **Anti-Patterns** -- list from research: replicating GSD state management, command-aware SKILL.md, over-engineering config, using --system-prompt instead of --append-system-prompt
    7. **Dependency Direction** -- gsd-ralph depends on GSD and Claude Code, never the reverse. Updates to GSD should flow through without breaking gsd-ralph.

    Keep it concise -- under 120 lines. This is a reference doc, not a design spec.
